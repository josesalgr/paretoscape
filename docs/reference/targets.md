# Targets API

Define feature-level targets for a planning problem.

Targets are stored in `x$data$targets` and later translated into linear
constraints when the optimization model is built. They are defined at
the feature level, but are evaluated against the total contribution
generated by selected actions in the final solution.

By default, all actions are allowed to contribute toward target
achievement. Optionally, the set of contributing actions can be
restricted through the `subset` argument.

## Details

Let \\\mathcal{P}\\ denote the set of planning units, \\\mathcal{A}\\
the set of actions, and \\\mathcal{F}\\ the set of features.

Let \\x\_{ia} \in \\0,1\\\\ denote whether action \\a \in \mathcal{A}\\
is selected in planning unit \\i \in \mathcal{P}\\.

Let \\c\_{iaf}\\ denote the contribution of action \\a\\ in planning
unit \\i\\ toward feature \\f\\, as interpreted by the model from the
stored input data.

A target for feature \\f\\ is stored as an absolute threshold \\T_f \ge
0\\, together with metadata describing how that threshold was defined.
When the optimization model is built, the target is translated into a
constraint of the form \$\$ \sum\_{(i,a) \in \mathcal{S}\_f} c\_{iaf}
x\_{ia} \ge T_f, \$\$ where \\\mathcal{S}\_f\\ is the set of planning
unit–action pairs allowed to count toward achievement of the target for
feature \\f\\.

By default, \\\mathcal{S}\_f\\ includes all feasible actions. If a
`subset` is supplied, then \\\mathcal{S}\_f\\ is restricted to the
actions matching that subset.

**Subset semantics**

The `subset` argument does *not* change the target value itself. It only
determines which actions are allowed to contribute toward target
achievement when the model is built.

A subset specification may contain:

- action identifiers from `x$data$actions$id`,

- action-set labels from `x$data$actions$action_set`,

- or a mixture of both.

**Stored fields**

Each stored target row contains, at minimum:

- `feature`:

  Feature identifier.

- `sense`:

  Constraint direction, currently `"ge"`.

- `target_unit`:

  How the target was defined, for example `"absolute"` or
  `"relative_baseline"`.

- `target_raw`:

  The user-supplied target value before conversion.

- `basis_total`:

  The baseline total used when converting a relative target into an
  absolute threshold.

- `target_value`:

  The final absolute threshold stored for model building.

- `subset`:

  A text representation of the action subset allowed to contribute.

- `label`:

  Optional user label.

- `created_at`:

  Creation timestamp.

**Absolute vs relative targets**

Absolute targets are stored directly as absolute thresholds: \$\$ T_f =
t_f, \$\$ where \\t_f\\ is the value supplied by the user.

Relative targets are supplied as proportions \\r_f \in \[0,1\]\\ and are
converted into absolute thresholds using the current baseline total
amount of each feature: \$\$ T_f = r_f \times B_f, \$\$ where \\B_f\\ is
the current total amount of feature \\f\\ in the landscape, computed by
`.pa_feature_totals()`.

**Overwrite behaviour**

If `overwrite = TRUE`, existing targets for the same feature and subset
specification may be replaced. If `overwrite = FALSE`, new target rows
are appended and any conflict handling is delegated to
`.pa_store_targets()`.

**Input format**

The `targets` argument is parsed internally by `.pa_parse_targets()` and
may typically be provided as:

- a single numeric value recycled to all features,

- a numeric vector aligned to feature order,

- a named numeric vector where names identify features,

- a `data.frame` with feature identifiers and target values.
